"Blog" File Format Specification

A unified file format for WAL, Blob, and related data in RocksDB

Status: Implementation in progress  |  Format name: "blog" (portmanteau of "blob" + "log")

File-Level Structure

A blog file consists of three regions: a fixed header with a property section, a body of records, and an optional footer composed of footer records.

+---------------------------------------------------------------------+ | FILE HEADER (40 bytes fixed) | | 96-bit magic, schema version, checksum type, compact record type, | | flags, escape sequence, reserved, property section size, checksum | +---------------------------------------------------------------------+ | HEADER PROPERTY SECTION | | Named key-value pairs + 5-byte trailer (compression + checksum) | | Followed by arbitrarily long padding (0x00 or 0xFF) | +---------------------------------------------------------------------+ | FILE BODY | | Sequence of records, each starting with the escape sequence. | | Record types: blob, WriteBatch, preamble-start, footer records. | +---------------------------------------------------------------------+ | FILE FOOTER (optional) | | Footer index records (0xF0), footer properties record (0xF8), | | file checksum info (0xFC, placeholder), locator (0xFF) last. | +---------------------------------------------------------------------+

Header Property Section

Immediately follows the fixed header. Contains only required (uppercase) named properties that the reader must recognize or reject the file. Ignorable (lowercase) properties are stored in separate ignorable properties records in the body. Encoded as:

[name_len: prefix_varint32] [name: name_len bytes] [value_len: prefix_varint32] [value: value_len bytes] ...

Followed by a 5-byte trailer: [compression_type: 1B][checksum: 4B] (same as block-based table). The checksum covers the property data + compression_type, with a context checksum modifier using the incarnation ID at offset kBlogFileFixedHeaderSize. The section may be empty (trailer only) when no required properties are set.

After the property section, arbitrarily long padding (0x00 or 0xFF, differing from the last meaningful byte) aligns the first record's escape sequence. This supports stronger alignment requirements (flush/sync boundaries) or recycling files of all zeros.

Escape Sequences

Decision: 10-byte escape sequence: 6 random bytes + 4 derived bytes. First byte not 0x00 nor 0xFF (distinguishable from padding). Bytes [6-9] = lower32(XXH3_64bits_withSeed(bytes[0-5], kBlogEscapeSeqSeed)), enabling heuristic detection without knowing the header.

Escape sequences must be 32-bit aligned in the file. The first 4 bytes (little-endian) serve as the incarnation ID for context checksums.

Primary Roles

  • WAL recycling detection: When recovery encounters undecodable data, scan for our escape sequence. If not found, the data is stale/recycled (different random sequence). The kBlogFileRecycled flag in the header indicates recycling is expected.
  • Non-linear recovery: After corruption, scan forward for the next escape sequence to resume recovery (kSkipAnyCorruptedRecords).
  • Record boundary discovery: Find record boundaries from any file position.

Format Selection

The record format is determined by the prefix-varint-encoded length field:

  • Value 0 (1 byte): Trivial format (no payload).
  • Values 1–16383 (1–2 bytes): Compact format.
  • 3+ byte prefix varint: Full format. Includes type, pre-payload compression type, and prefix checksum.
  • length = 263 − 1: Unspecified size (full format). Reader scans forward for next escape sequence.

Trivial Record Format (prefix varint value 0)

[escape_seq: 10B] [prefix_varint(0): 1B] [type: 1B] [checksum: 4B] [padding: 0+ B]

For records with no payload (e.g. preamble-start). The checksum covers (prefix_varint + type), with a context checksum modifier. No compression type byte since there is no payload to compress.

Compact Record Format (prefix varint 1–2 bytes, values 1–16383)

[escape_seq: 10B] [prefix_varint length: 1-2B] [payload: length bytes] [compression_type: 1B] [checksum: 4B] [padding: 0+ B]

No type byte — type comes from the header's compact_record_type field. Used for payloads < 16 KiB. The compact format has no separate length checksum; the checksum in the trailer covers the payload and compression_type byte with a context modifier derived from the record's file offset and incarnation ID. An incorrect length field will cause the checksum to fail because the payload bytes fed to the checksum will be wrong. Adding a type byte to the compact format would require including it in the checksum computation (or modifying the escape sequence) to maintain integrity.

Full Record Format (prefix varint ≥ 3 bytes)

[escape_seq: 10B] [prefix_varint length: 3+B] [type: 1B] [compression_type: 1B] [prefix_checksum: 4B] [payload: length bytes] [compression_type: 1B] [checksum: 4B] [padding: 0+ B]

The prefix_checksum covers (prefix_varint + type + compression_type), with a context checksum modifier for defense-in-depth. Validates the length before attempting a potentially large read. The pre-payload compression_type enables streaming decompression without seeking to the trailer.

Records that don't match the compact_record_type (e.g. footer records) always use full format by encoding the prefix varint with ≥ 3 bytes (irregular encoding), but length 0 is disallowed and must use the trivial record format.

All checksums in blog files (fixed header, property section trailer, record prefix, record trailer) use context checksum modifiers incorporating the incarnation ID, providing defense-in-depth for misplaced data detection even with kNoChecksum.

Compression

Per-Record Compression

Each record's trailer compression_type byte indicates the compression algorithm used for that record's payload. The actual type comes from Compressor::CompressBlock output, not the configured type. This is the same approach used by SST block-based table builders.

Streaming Compression (WriteBatch records only)

When the header property WriteBatchStreamingCompressionType is set, WriteBatch record payloads may use streaming compression across records. Such records use kStreamingCompressionSentinel (0x7F) as their trailer's compression_type byte, indicating "compressed using the streaming algorithm declared in the header."

A record with kNoCompression (0x00) in the trailer remains individually uncompressed even when streaming is active, enabling dynamic on/off switching based on compression effectiveness.

Non-WriteBatch records (blob records, footer records) are always outside the streaming context and use per-record compression, keeping blob data randomly readable.

Currently only ZSTD is supported for streaming compression. The WriteBatchStreamingCompressionType property is required (uppercase first letter), so older readers that don't understand it will reject the file rather than misinterpret streaming-compressed payloads.

Padding Scheme

Padding fills the gap between a record's trailer (or the header property section) and the next escape sequence. The padding byte value is chosen to differ from the last meaningful byte, enabling unambiguous scanning.

Last meaningful bytePad withMandatory?
0x000xFFYes -- at least 1 byte required
0xFF0x00Yes -- at least 1 byte required
Anything else0x00 preferred; 0xFF also legalOnly if needed for alignment

Padding may be arbitrarily long for stronger alignment requirements (for example, rounding up to 512-byte or 4 KiB flush/sync-friendly boundaries) or for recycling a file of all zeros. The reader skips padding by scanning forward over 0x00/0xFF bytes in 4-byte-aligned chunks until finding a byte that is neither, which is the first byte of the next escape sequence (guaranteed to be neither 0x00 nor 0xFF).

Context Checksums

Decision (adopted from SST format_version=6): Each record's checksum incorporates a 32-bit incarnation ID (first 4 bytes of the escape sequence). Uses ChecksumModifierForContext(incarnation_id, record_offset) -- the same mechanism as block-based SST files.

Context checksums provide defense-in-depth for recycling detection and general data integrity verification with zero per-record overhead.

The checksum type is configured via DBOptions::blog_checksum. The same option is intended to govern blog-format internal checksums across file roles.

Footer

The footer is composed of regular records using the full record format, maintaining simple forward parsability. A file is cleanly sealed when a valid footer locator record is the last record. Footer records appear in the order listed below.

  1. Footer index records (0xF0): Large metadata like sparse indexes for range queries. Zero or more per file.
  2. Footer properties record (0xF8): Named properties (same encoding as header properties, but the full record format's own trailer provides integrity). At most one per file.
  3. Footer file checksum info (0xFC): Placeholder for future use. Would contain a dump of the full file checksum state, allowing the file checksum to be inferred from just reading the tail of the file.
  4. Footer locator record (0xFF): Must be the last record. Contains relative offsets (in 4-byte units) to other footer records, sequential relative to each other in reverse order.

Footer Locator Payload

[record_type: 1B] [relative_offset_in_4B_units: uint32 LE] ...

First entry: distance from locator's escape_seq backward to the preceding footer record. Subsequent: from the previous entry backward to the next.

Exact physical file size, when known, is tracked separately in blob MANIFEST metadata.

Blob MANIFEST Metadata

Blob files are tracked in MANIFEST via BlobFileAddition records. In addition to the long-standing blob count/byte totals and file checksum fields, the current blob implementation persists the following per-file metadata for format-aware accounting and verification.

FieldEncoding / When PresentPurpose
physical_blob_file_sizeForward-compatible custom field. Varint64 actual file size. Emitted when known for both legacy and blog blob files.Provides an exact physical size for verification and accounting. This avoids relying on format-specific estimates from total_blob_bytes plus hard-coded header/footer constants.
format descriptorForward-incompatible custom field. Emitted only for non-legacy blog blob files. Payload: [schema_version: 1B][file_identity: 10B].Makes the blob file format explicit on a per-file basis and ensures older readers fail fast rather than silently misinterpreting a blog blob file.
schema_versionStored inside the format descriptor. Manifest value 0 is reserved to mean legacy blob log format by omission of the descriptor. Current blog blob files use schema version 1.Determines how blob bytes are interpreted for mixed legacy/blog deployments and future schema evolution.
file_identityStored inside the format descriptor. Exactly the blog file header's 10-byte escape sequence.Provides a cheap but precise check that the MANIFEST entry refers to the same physical blob file that is being opened.

On open, RocksDB verifies this metadata against the actual blob file: physical size, legacy-vs-blog format, blog schema version, and file identity must all match.

Motivation:

  • Mixed legacy/blog blob files can coexist in one DB across option flips or upgrades, so format interpretation must be per-file rather than inferred from a DB-wide setting.
  • total_blob_bytes intentionally tracks blob payload plus record framing, not an exact file length. Persisting the physical file size removes avoidable approximation error from header/footer differences.
  • Using the escape sequence as MANIFEST-side file identity catches stale or mismatched files early during recovery and verification without inventing a second file-identity mechanism.

WAL Recovery Modes

The escape sequence is the primary mechanism for detecting WAL recycling and for non-linear data recovery. The kBlogFileRecycled flag in the header indicates the file may contain trailing stale data from a previous incarnation.

Mode Behavior on Undecodable Data Escape Sequence Role
kAbsoluteConsistency Error on any failure, including incomplete trailing records. If recycled and no escape sequence found in trailing data, accept as clean end. Distinguishes stale recycled data from corruption.
kTolerateCorruptedTailRecords Incomplete trailing record (Incomplete status): tolerate, stop at last complete record. Corruption: scan for escape sequence. Not found = clean end (recycled or preallocated). Found with valid record beyond = mid-file corruption (error). Distinguishes tail damage from mid-file corruption.
kPointInTimeRecovery (default) Stop at first inconsistency. Incomplete trailing records tolerated. Recycled trailing data accepted. Diagnostic improvement.
kSkipAnyCorruptedRecords On corruption, scan forward for next escape sequence. If found, try to decode and verify; resume if valid. If not found, stop. Essential for record discovery after corruption.

WriteBatch Blob Preamble

A blob preamble is a sequence of blob records preceded by a preamble-start record (type 0x11, trivial format, length=0) and followed by the associated WriteBatch record. The blob records contain separated values referenced by file_number = 0 blob references in the WriteBatch.

This functionality is stubbed in the current implementation: the preamble-start record type is defined and can be written/read, but the full preamble flow (in-memory serialization, blob reference encoding) is deferred.

Ignorable Properties Records

Ignorable properties records (kBlogIgnorablePropertiesRecord, type 0x01) carry diagnostic/debugging named properties using the same encoding as the header property section. They may appear anywhere in the file body:

  • Initial writer: immediately after the header, the writer emits an ignorable properties record for its diagnostic properties (dbSessionId, dbHostId, creator, creationTime, role, etc.). The header property section itself contains only required/uppercase properties.
  • File reuse (e.g. manifest reuse): when an existing file is re-opened for appending by a new writer (potentially on a different host or RocksDB version), the new writer emits another ignorable properties record with its own identity and diagnostics. This is why multiple ignorable properties records may appear in a single file.

Restrictions:

  • Only ignorable (lowercase) properties are allowed in these records. Required (uppercase) properties are set once in the header property section and cannot appear here.
  • Readers may skip these records entirely without affecting correctness.

The upper/lowercase naming convention remains useful for distinguishing required vs. ignorable properties when they are combined into a single map or dictionary for inspection.

Replication Log Column Family

Deferred. The blog format supports an "rlog" role hint for replication log storage, with sparse indexes in footer records and in-memory index maintenance. See design notes in the original proposal.

DB Options

  • use_blog_format_for_wals (bool, default false): New WAL files use blog format.
  • use_blog_format_for_blobs (bool, default false): New blob files use blog format.
  • blog_checksum (ChecksumType, default kXXH3): Checksum algorithm for blog format files. ChecksumType enum moved to include/rocksdb/checksum_type.h (from table.h) for use by both table and DB options.

All are immutable DBOptions. Existing legacy WAL and blob files are always readable regardless of these settings.

Compression Manager Integration

  • Blog blob files use the CF's configured compression manager (via MutableCFOptions::compression_manager). Legacy blob files continue to use the builtin manager for backward compatibility.
  • Blog WAL files use per-record compression via Compressor::CompressBlock. The actual CompressionType from CompressBlock output is stored in the record trailer.
  • Streaming compression infrastructure exists (WriteBatchStreamingCompressionType property, kStreamingCompressionSentinel 0x7F trailer value) but is not yet enabled by default.
