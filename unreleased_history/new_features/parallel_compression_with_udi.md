Parallel compression (`CompressionOptions::parallel_threads > 1`) is now supported when a `user_defined_index_factory` is configured, provided each `IndexFactoryBuilder` opts in by overriding `SupportsParallelAddEntry()` and the parallel `PrepareAddEntry` / `FinishAddEntry` protocol. The built-in trie index (`trie_index::TrieIndexFactory`) opts in. Custom factories that have not opted in keep the previous single-threaded behavior -- the table builder silently falls back to `parallel_threads=1` for the whole SST when any configured builder reports `SupportsParallelAddEntry()==false`. The `CompressionOptions::parallel_threads` value itself is no longer sanitized to 1 by the UDI presence check (it is still sanitized to 1 by the unrelated `partition_filters && !decouple_partitioned_filters` constraint).
