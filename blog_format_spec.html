"Blog" File Format Specification

A unified file format for WAL, Blob, and related data in RocksDB

Status: Implementation in progress  |  Format name: "blog" (portmanteau of "blob" + "log")

File-Level Structure

A blog file consists of three regions: a fixed header with a property section, a body of records, and an optional footer composed of footer records.

+---------------------------------------------------------------------+ | FILE HEADER (40 bytes fixed) | | 96-bit magic, schema version, checksum type, compact record type, | | flags, escape sequence, reserved, property section size, checksum | +---------------------------------------------------------------------+ | HEADER PROPERTY SECTION | | Named key-value pairs + 5-byte trailer (compression + checksum) | | Followed by arbitrarily long padding (0x00 or 0xFF) | +---------------------------------------------------------------------+ | FILE BODY | | Sequence of records, each starting with the escape sequence. | | Record types: blob, WriteBatch, preamble-start, footer records. | +---------------------------------------------------------------------+ | FILE FOOTER (optional) | | Footer index records (0xFD), footer properties record (0xFE), | | footer locator record (0xFF) -- must be last. | +---------------------------------------------------------------------+

Header Property Section

Immediately follows the fixed header. Contains named key-value pairs encoded as:

[name_len: varint32] [name: name_len bytes] [value_len: varint32] [value: value_len bytes] ...

Followed by a 5-byte trailer: [compression_type: 1B][checksum: 4B] (same as block-based table). The checksum covers the property data + compression_type, with a context checksum modifier using the incarnation ID at offset kBlogFileFixedHeaderSize.

After the property section, arbitrarily long padding (0x00 or 0xFF, differing from the last meaningful byte) aligns the first record's escape sequence. This supports stronger alignment requirements (flush/sync boundaries) or recycling files of all zeros.

Escape Sequences

Decision: 10-byte escape sequence: 6 random bytes + 4 derived bytes. First byte not 0x00 nor 0xFF (distinguishable from padding). Bytes [6-9] = lower32(XXH3_64bits_withSeed(bytes[0-5], kBlogEscapeSeqSeed)), enabling heuristic detection without knowing the header.

Escape sequences must be 32-bit aligned in the file. The first 4 bytes (little-endian) serve as the incarnation ID for context checksums.

Primary Roles

  • WAL recycling detection: When recovery encounters undecodable data, scan for our escape sequence. If not found, the data is stale/recycled (different random sequence). The kBlogFileRecycled flag in the header indicates recycling is expected.
  • Non-linear recovery: After corruption, scan forward for the next escape sequence to resume recovery (kSkipAnyCorruptedRecords).
  • Record boundary discovery: Find record boundaries from any file position.

Length Semantics

  • length = 0: No payload and no 5-byte trailer. Record is escape_seq + prefix fields + padding.
  • length > 0: Payload of length bytes followed by 5-byte trailer.
  • length = 2^63 - 1: Unspecified size. Reader scans forward for next escape sequence.

Compact Record Format (varint length ≤ 2 bytes, default record type)

[escape_seq: 10B] [varint length: 1-2B] [payload: length bytes] [compression_type: 1B] [checksum: 4B] [padding: 0+ B]

No type byte -- type comes from the header's compact_record_type field. Used for payloads < 16 KiB.

Full Record Format (varint length > 2 bytes, or forced)

[escape_seq: 10B] [varint length: 3+B] [type: 1B] [compression_type: 1B] [prefix_checksum: 4B] [payload: length bytes] (if length > 0) [compression_type: 1B] [checksum: 4B] (if length > 0) [padding: 0+ B]

The prefix_checksum covers (varint + type + compression_type), with a context checksum modifier for defense-in-depth. Validates the length before attempting a potentially large read. The pre-payload compression_type enables streaming decompression without seeking to the trailer.

Records that don't match the compact_record_type (e.g. footer records, preamble-start) always use full format by encoding the varint with ≥ 3 bytes (irregular encoding).

All checksums in blog files (fixed header, property section trailer, record prefix, record trailer) use context checksum modifiers incorporating the incarnation ID, providing defense-in-depth for misplaced data detection even with kNoChecksum.

Compression

Per-Record Compression

Each record's trailer compression_type byte indicates the compression algorithm used for that record's payload. The actual type comes from Compressor::CompressBlock output, not the configured type. This is the same approach used by SST block-based table builders.

Streaming Compression (WriteBatch records only)

When the header property WriteBatchStreamingCompressionType is set, WriteBatch record payloads may use streaming compression across records. Such records use kStreamingCompressionSentinel (0x7F) as their trailer's compression_type byte, indicating "compressed using the streaming algorithm declared in the header."

A record with kNoCompression (0x00) in the trailer remains individually uncompressed even when streaming is active, enabling dynamic on/off switching based on compression effectiveness.

Non-WriteBatch records (blob records, footer records) are always outside the streaming context and use per-record compression, keeping blob data randomly readable.

Currently only ZSTD is supported for streaming compression. The WriteBatchStreamingCompressionType property is required (uppercase first letter), so older readers that don't understand it will reject the file rather than misinterpret streaming-compressed payloads.

Padding Scheme

Padding fills the gap between a record's trailer (or the header property section) and the next escape sequence. The padding byte value is chosen to differ from the last meaningful byte, enabling unambiguous scanning.

Last meaningful bytePad withMandatory?
0x000xFFYes -- at least 1 byte required
0xFF0x00Yes -- at least 1 byte required
Anything else0x00 (writer's choice)Only if needed for alignment

Padding may be arbitrarily long for stronger alignment requirements (e.g. flush/sync boundaries) or for recycling a file of all zeros. The reader skips padding by scanning forward over 0x00/0xFF bytes in 4-byte-aligned chunks until finding a byte that is neither, which is the first byte of the next escape sequence (guaranteed to be neither 0x00 nor 0xFF).

Context Checksums

Decision (adopted from SST format_version=6): Each record's checksum incorporates a 32-bit incarnation ID (first 4 bytes of the escape sequence). Uses ChecksumModifierForContext(incarnation_id, record_offset) -- the same mechanism as block-based SST files.

Context checksums provide defense-in-depth for recycling detection and general data integrity verification with zero per-record overhead.

The checksum type is configured from the CF's BlockBasedTableOptions::checksum setting (for blob files) or the default CF's setting (for WAL files). This is a reasonable but hopefully temporary way of configuring blog file internal checksums; a dedicated option may be added later.

Footer

The footer is composed of regular records using the full record format, maintaining simple forward parsability. A file is cleanly sealed when a valid footer locator record is the last record. Footer records appear in the order listed below.

  1. Footer index records (0xF0): Large metadata like sparse indexes for range queries. Zero or more per file.
  2. Footer properties record (0xF8): Named properties (same encoding as header properties, but the full record format's own trailer provides integrity). At most one per file.
  3. Footer file checksum info (0xFC): Placeholder for future use. Would contain a dump of the full file checksum state, allowing the file checksum to be inferred from just reading the tail of the file.
  4. Footer locator record (0xFF): Must be the last record. Contains relative offsets (in 4-byte units) to other footer records, sequential relative to each other in reverse order.

Footer Locator Payload

[record_type: 1B] [relative_offset_in_4B_units: uint32 LE] ...

First entry: distance from locator's escape_seq backward to the preceding footer record. Subsequent: from the previous entry backward to the next.

WAL Recovery Modes

The escape sequence is the primary mechanism for detecting WAL recycling and for non-linear data recovery. The kBlogFileRecycled flag in the header indicates the file may contain trailing stale data from a previous incarnation.

Mode Behavior on Undecodable Data Escape Sequence Role
kAbsoluteConsistency Error on any failure, including incomplete trailing records. If recycled and no escape sequence found in trailing data, accept as clean end. Distinguishes stale recycled data from corruption.
kTolerateCorruptedTailRecords Incomplete trailing record (Incomplete status): tolerate, stop at last complete record. Corruption: scan for escape sequence. Not found = clean end (recycled or preallocated). Found with valid record beyond = mid-file corruption (error). Distinguishes tail damage from mid-file corruption.
kPointInTimeRecovery (default) Stop at first inconsistency. Incomplete trailing records tolerated. Recycled trailing data accepted. Diagnostic improvement.
kSkipAnyCorruptedRecords On corruption, scan forward for next escape sequence. If found, try to decode and verify; resume if valid. If not found, stop. Essential for record discovery after corruption.

WriteBatch Blob Preamble

A blob preamble is a sequence of blob records preceded by a preamble-start record (type 0x02, full format, length=0) and followed by the associated WriteBatch record. The blob records contain separated values referenced by file_number = 0 blob references in the WriteBatch.

This functionality is stubbed in the current implementation: the preamble-start record type is defined and can be written/read, but the full preamble flow (in-memory serialization, blob reference encoding) is deferred.

Replication Log Column Family (ShallowL0Store)

Deferred. The blog format supports a "rlog" role hint for replication log storage, with sparse indexes in footer records and in-memory index maintenance. See design notes in the original proposal.

DB Options

  • use_blog_format_for_wals (bool, default false): New WAL files use blog format.
  • use_blog_format_for_blobs (bool, default false): New blob files use blog format.
  • blog_checksum (ChecksumType, default kXXH3): Checksum algorithm for blog format files. ChecksumType enum moved to include/rocksdb/checksum_type.h (from table.h) for use by both table and DB options.

All are immutable DBOptions. Existing legacy WAL and blob files are always readable regardless of these settings.

Compression Manager Integration

  • Blog blob files use the CF's configured compression manager (via MutableCFOptions::compression_manager). Legacy blob files continue to use the builtin manager for backward compatibility.
  • Blog WAL files use per-record compression via Compressor::CompressBlock. The actual CompressionType from CompressBlock output is stored in the record trailer.
  • Streaming compression infrastructure exists (WriteBatchStreamingCompressionType property, kStreamingCompressionSentinel 0x7F trailer value) but is not yet enabled by default.
