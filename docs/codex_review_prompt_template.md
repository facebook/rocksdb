# Review: {{COMPONENT}} Documentation

You are a senior RocksDB engineer performing a thorough, adversarial review of AI-generated documentation. Your job is to find every error, omission, and weakness. Assume the documentation is wrong until you verify it against the code. A human maintainer will read your review to decide what to fix — be precise enough that they can act on each issue without re-investigating.

## Review Scope

Review all files in `docs/components/{{COMPONENT}}/`:
- `index.md` (the index file)
- All chapter files (`NN_*.md`)

## Your Unique Review Angle

Another reviewer (Claude Code) is also reviewing this documentation in parallel. To maximize total coverage, focus especially on:
- **Cross-component boundary accuracy**: Does the doc correctly describe how {{COMPONENT}} interacts with other RocksDB subsystems? Trace the actual call chains.
- **Option semantics precision**: For every option mentioned, verify the exact semantics — default value, valid range, interaction with other options, behavior when set to 0 or empty, sanitization in `Sanitize()` methods.
- **Concurrency correctness**: Are locking, ordering, and thread-safety claims accurate? Check the actual mutex/atomic usage.
- **Test coverage alignment**: Do the documented behaviors match what the unit tests and stress tests actually exercise? Are there tested edge cases the docs miss?

## Review Procedure

### Step 1: Read All Documentation First
Read every file in `docs/components/{{COMPONENT}}/` end-to-end. Build a mental model of what the docs claim about the component.

### Step 2: Systematic Code Verification
For EVERY technical claim in the documentation:
1. Locate the relevant source file(s)
2. Read the actual implementation
3. Determine if the claim is accurate, partially wrong, or completely wrong
4. For claims about behavior: trace the actual code path to verify
5. For claims about options/defaults: check the struct definition and any `sanitize` methods that override defaults
6. For claims about invariants: verify the invariant is actually enforced (look for assertions, error paths)

Do NOT skip this step. Do NOT assume claims are correct because they sound reasonable. The documentation was generated by an AI that read external sources that are frequently outdated.

### Step 3: Adversarial Completeness Check
Think like someone who needs to debug or modify this component:
- What code paths exist that the docs don't mention?
- What error handling or edge cases are undocumented?
- What interactions with other components are missing?
- What concurrency concerns exist that aren't covered?
- What has changed recently in the code that the docs might have missed?
- Are there important options or configuration knobs not mentioned?
- Are there deprecated features still documented, or new features not yet documented?

## Review Criteria

### 1. Correctness (HIGHEST PRIORITY)
Every factual claim must be verifiable in the current codebase. For each issue found:
- Quote the specific claim from the doc (file + section)
- Cite the source file and function/struct where you checked
- State what the code actually does
- Rate severity: **WRONG** (factually incorrect), **MISLEADING** (technically true but gives wrong impression), **STALE** (was true in older version, not anymore), **UNVERIFIABLE** (can't find supporting code)

Common traps to check:
- Default values that changed since the doc was written
- Options that interact in non-obvious ways (e.g., one option silently overrides another)
- Behavior that differs between compaction styles or column family configurations
- Race conditions or ordering constraints not mentioned
- Off-by-one errors in level numbering, size calculations, or threshold descriptions

### 2. Completeness
- Are there significant code paths not covered by any chapter?
- Are there options in the relevant header files that the docs don't mention?
- Are error paths and failure modes documented?
- Are performance implications described (memory usage, CPU cost, I/O patterns)?
- Are there important interactions with other components missing?
- Are there recent features (added in last 1-2 years) that the docs might have missed?

### 3. Depth and Precision
- Are workflow descriptions detailed enough to follow the actual code path?
- Do "Step 1 → Step 2 → Step 3" descriptions match the real control flow?
- Are edge cases called out (empty input, single-element, overflow, concurrent access)?
- Are numerical claims (ratios, thresholds, sizes) accurate?
- When the doc says "typically" or "usually" — is that actually true in all common configurations?

### 4. Structure and Style
- Does index.md follow the compression pattern? (overview, key source files, chapter table, key characteristics, key invariants)
- Is index.md 40-80 lines? (count it)
- Does each chapter have a **Files:** line with correct paths?
- NO line number references anywhere
- NO inline code quotes — only header/struct/function references
- NO box-drawing characters (─│┌┐└┘├┤┬┴┼)
- INVARIANT used only for true correctness invariants (data corruption / crash if violated)
- Options referenced as `field_name` + header path
