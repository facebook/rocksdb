# Operation Mix

**Files:** `db_stress_tool/db_stress_test_base.cc`, `db_stress_tool/db_stress_gflags.cc`, `db_stress_tool/db_stress_common.h`

## Operation Selection

The main operation loop runs in `StressTest::OperateDb()` (see `db_stress_tool/db_stress_test_base.cc`). For each of `--ops_per_thread` iterations, the thread selects an operation based on configurable percentage flags.

### Percentage-Based Operations

These flags must sum to exactly 100 (validated at startup in `db_stress_tool.cc`):

| Flag | Default | Operation |
|------|---------|-----------|
| `--readpercent` | 45 | `TestGet()` -- point lookup |
| `--prefixpercent` | 5 | `TestPrefixScan()` -- prefix iteration |
| `--writepercent` | 35 | `TestPut()` -- write (Put, PutEntity, or Merge) |
| `--delpercent` | 4 | `TestDelete()` -- delete or single delete |
| `--delrangepercent` | 1 | `TestDeleteRange()` -- range deletion |
| `--iterpercent` | 10 | `TestIterate()` -- forward/backward iteration |
| `--customopspercent` | 0 | `TestCustomOperations()` -- mode-specific operations (used by MultiOpsTxns) |

The operation is selected by drawing a uniform random number in [0, 100) and mapping it to the cumulative ranges defined by these percentages.

### Probabilistic Operations

In addition to the percentage-based main operation, each iteration may trigger one or more probabilistic side operations. These are controlled by `_one_in` flags, where a flag value of N means the operation occurs with probability 1/N:

| Flag | Operation |
|------|-----------|
| `--flush_one_in` | `Flush()` |
| `--compact_range_one_in` | `CompactRange()` |
| `--compact_files_one_in` | `CompactFiles()` |
| `--checkpoint_one_in` | Create a checkpoint and optionally open/verify it |
| `--backup_one_in` | Create and verify a backup |
| `--acquire_snapshot_one_in` | Acquire a snapshot and record current state |
| `--set_options_one_in` | Call `SetOptions()` with random dynamic option values |
| `--ingest_external_file_one_in` | Build and ingest an external SST file |
| `--pause_background_one_in` | Temporarily pause background operations |
| `--disable_file_deletions_one_in` | Temporarily disable file deletions |
| `--disable_manual_compaction_one_in` | Temporarily disable manual compaction |
| `--verify_db_one_in` | Run `VerifyDb()` inline |
| `--verify_checksum_one_in` | Call `VerifyChecksum()` |
| `--get_property_one_in` | Call `GetProperty()` and log results |
| `--reset_stats_one_in` | Call `ResetStats()` |
| `--key_may_exist_one_in` | Call `KeyMayExist()` |
| `--lock_wal_one_in` | Call `LockWAL()` / `UnlockWAL()` |
| `--manual_wal_flush_one_in` | Call `FlushWAL()` |
| `--get_live_files_apis_one_in` | Call `GetLiveFiles()`, `GetLiveFilesMetaData()`, etc. |

## Key Generation

### Key Space

The key space ranges from 0 to `--max_key - 1`. Keys are generated by `GenerateOneKey()` (see `db_stress_tool/db_stress_common.h`), which selects a random key within the thread's assigned portion of the key space.

### Active Window

The `--active_width` flag limits the range of keys a thread operates on at any given time. This creates temporal locality, making it more likely for threads to hit recently written data and exercise the memtable and L0 code paths.

### Key Encoding

Keys are encoded as fixed-width strings via the `Key()` function. The key format supports variable-length keys controlled by `--max_key_len` and `--key_len_percent_dist`, which define a distribution over key lengths to test different key sizes.

### Hot Key Distribution

`--hot_key_alpha` controls Zipfian distribution for key selection. When set to a positive value, frequently accessed keys are selected more often, exercising cache behavior and contention on hot keys.

## Write Operations

### Put Variants

When the selected operation is a write (`--writepercent`), the actual write type is selected based on additional flags:

| Condition | Write Type |
|-----------|-----------|
| `--use_put_entity_one_in > 0` and matches | `PutEntity()` with wide columns |
| `--use_timed_put_one_in > 0` and matches | `TimedPut()` with write unix time |
| `--use_merge` is set | `Merge()` |
| Default | `Put()` |

### Value Generation

Values are generated by `GenerateValue()` which encodes the value base into the value bytes. During verification, `GetValueBase()` extracts the value base from a DB-returned value for comparison against the expected state.

## Column Family Selection

Each operation randomly selects a column family from the available CFs (0 to `--column_families - 1`). In some test modes, the `GenerateColumnFamilies()` method may return multiple CFs for operations that span CFs (e.g., CfConsistency mode).

### Column Family Clearing

When `--clear_column_family_one_in > 0`, the test periodically drops and re-creates a column family. This exercises the CF lifecycle but is incompatible with expected state persistence (`--expected_values_dir`) and backup operations.

## Dynamic Option Changes

When `--set_options_one_in > 0`, the test periodically calls `SetOptions()` with randomly selected values from a pre-built options table. The table is built by `StressTest::BuildOptionsTable()` (see `db_stress_tool/db_stress_test_base.cc`), which generates valid value ranges for dynamic options like `write_buffer_size`, `max_write_buffer_number`, `level0_slowdown_writes_trigger`, `compression_type`, and others.

## DB Reopen

When `--reopen > 0`, the test periodically closes and reopens the database (approximately every `ops_per_thread / reopen` operations). This exercises the recovery path and options loading. A voting mechanism ensures all threads agree to reopen simultaneously.

Note: Reopen is disabled when `--disable_wal=1` (unsafe) and in blackbox crash test mode (where external kills provide the crash-reopen cycle).
