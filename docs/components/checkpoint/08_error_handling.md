# Error Handling and Recovery

**Files:** `utilities/checkpoint/checkpoint_impl.cc`, `utilities/backup/backup_engine.cc`, `include/rocksdb/utilities/backup_engine.h`

## Checkpoint Failure Handling

If `CreateCheckpoint()` fails at any point during the workflow:

Step 1 -- The staging directory (`.tmp` suffix) and any files created in it are cleaned up via `CleanStagingDirectory()`.

Step 2 -- `EnableFileDeletions()` is called if `DisableFileDeletions()` succeeded earlier.

Step 3 -- The error status is returned to the caller.

`CleanStagingDirectory()` performs best-effort cleanup: it iterates over all files in the staging directory, deletes each one, then deletes the directory itself. If cleanup fails (e.g., permission denied), the error is logged but not propagated. A partial `.tmp` directory may remain on disk in rare failure scenarios and will be cleaned up automatically on the next `CreateCheckpoint()` call to the same path.

### Common Checkpoint Failure Modes

| Error | Cause |
|-------|-------|
| `InvalidArgument("Directory exists")` | `checkpoint_dir` already exists |
| `InvalidArgument("invalid checkpoint directory name")` | Empty path or root-only path |
| `NotSupported` | `db_paths` or `cf_paths` in use (multiple data directories) |
| `IOError` | Disk full, permission denied, filesystem error during link/copy |
| `Corruption` | Inconsistent file metadata (e.g., CURRENT file size mismatch) |
| `Aborted` | Failed to clean leftover staging directory |

## BackupEngine Failure Handling

### Incomplete Backup Detection

BackupEngine detects corrupt backups during `Open()`:

- Backup IDs are discovered by enumerating files in the `meta/` directory. For each metadata file, if parsing fails (e.g., corrupt content, mismatched checksums), the backup is added to the `corrupt_backups_` list.
- Orphaned files in `private/` or `shared_checksum/` with no corresponding metadata file are not represented as backup IDs -- they are simply garbage for GC to clean up.
- `Open()` sets the `might_need_garbage_collect_` flag but does NOT automatically run garbage collection.
- Corrupt backups are excluded from `GetBackupInfo()` results and listed by `GetCorruptedBackups()`.

### Garbage Collection

`GarbageCollect()` (see `BackupEngineAppendOnlyBase` in `include/rocksdb/utilities/backup_engine.h`) cleans up orphaned files:

- Files from incomplete backups (no corresponding metadata file)
- Orphaned files in `shared_checksum/` not referenced by any backup metadata
- Temporary files from crashed or interrupted operations

Garbage collection is triggered conditionally: `PurgeOldBackups()` and `DeleteBackup()` call `GarbageCollect()` only when `might_need_garbage_collect_` is set. `CreateNewBackup()` calls `GarbageCollect()` only when it finds a leftover private directory for the next backup ID.

Important: `GarbageCollect()` can delete files that would be needed to manually recover a corrupt backup or to preserve backups created by a newer, incompatible RocksDB version. Use with care.

### Backup Creation Failures

If `CreateNewBackup()` fails mid-operation:

- Partial files may remain in `shared_checksum/` or `private/<backup_id>/`
- The metadata file is NOT written (since metadata is written atomically only after all files are copied successfully)
- The backup directory remains consistent: no valid backup references the partial files
- Cleanup occurs automatically on the next `CreateNewBackup()`, `PurgeOldBackups()`, `DeleteBackup()`, or explicit `GarbageCollect()` call

### Backup Deletion Failures

If `DeleteBackup()` or `PurgeOldBackups()` fails mid-operation:

- The state will be cleaned up on the next call to `DeleteBackup()`, `PurgeOldBackups()`, or `GarbageCollect()`
- Shared files are reference-counted: a shared file is only deleted when no remaining backup references it

## ExportColumnFamily Failure Handling

If `ExportColumnFamily()` fails:

- All files in the staging directory (or final directory, if the rename already succeeded) are deleted
- The directory itself is deleted
- Cleanup errors are logged as warnings but do not affect the returned error status
- The `ExportImportFilesMetaData` output pointer is not modified on failure

## Session ID Collision Safety

With `kUseDbSessionId` naming (see `BackupEngineOptions::ShareFilesNaming` in `include/rocksdb/utilities/backup_engine.h`), shared file names include the DB session ID generated by `SemiStructuredUniqueIdGen` (a 128-bit counter-based generator initialized from random state). This provides strong uniqueness guarantees without requiring coordination between backup operations.

For SST files created by older RocksDB versions that do not include a session ID, the naming automatically falls back to `kLegacyCrc32cAndFileSize`, which uses the file's CRC32C checksum and size for naming.
