# VersionStorageInfo

**Files:** `db/version_set.h`, `db/version_set.cc`

## Overview

`VersionStorageInfo` holds the complete set of SST files and blob files for one column family at one point in time. It is the storage-layer portion of a `Version` and contains both the raw file data and derived structures used for compaction scheduling and file lookup.

## Primary Data Structures

| Field | Description |
|-------|-------------|
| `files_[level]` | `vector<FileMetaData*>` per level -- the SST files at each level |
| `blob_files_` | `vector<shared_ptr<BlobFileMetaData>>` sorted by blob file number |
| `level_files_brief_[level]` | `LevelFilesBrief` -- compact array of `FdWithKeyRange` for fast lookup |
| `file_locations_` | `UnorderedMap<uint64_t, FileLocation>` -- maps file number to (level, position) |
| `compact_cursor_[level]` | Round-robin compaction cursor per level |

## Derived Structures

These are generated by `PrepareForVersionAppend()` and must be computed before `SetFinalized()` is called:

| Structure | Generator | Purpose |
|-----------|-----------|---------|
| Compensated sizes | `ComputeCompensatedSizes()` | Adjusts file sizes for deletion density |
| Non-empty level count | `UpdateNumNonEmptyLevels()` | Tracks how many levels have files |
| Base bytes | `CalculateBaseBytes()` | Computes dynamic level targets |
| Compaction priority | `UpdateFilesByCompactionPri()` | Sorts files by compaction urgency |
| File indexer | `GenerateFileIndexer()` | Cross-level file overlap index |
| Level files brief | `GenerateLevelFilesBrief()` | Compact read-optimized file arrays |
| L0 non-overlapping flag | `GenerateLevel0NonOverlapping()` | Detects non-overlapping L0 |
| Bottommost files | `GenerateBottommostFiles()` | Identifies files at effective bottom |
| File location index | `GenerateFileLocationIndex()` | Maps file numbers to locations |

Note: Compaction scores are computed separately via `ComputeCompactionScore()` called from `VersionSet::AppendVersion()`, not from `PrepareForVersionAppend()`.

## Compaction Scoring

`ComputeCompactionScore()` calculates compaction urgency for each level. Scoring depends on compaction style:

- **L0**: Score is the maximum of file-count pressure (file count / `level0_file_num_compaction_trigger`) and size pressure (total size / max_bytes_for_level_base). For universal compaction, `CalculateBaseBytes()` folds non-empty lower levels into a total sorted-run count, and that count feeds the L0-style score and stall logic. The `l0_delay_trigger_count_` tracks the count used for write stall decisions, which may be the sorted-run count (universal) rather than the raw L0 file count.
- **L1+** (level compaction): Score = level_size / target_size, with adjustments for dynamic-level-bytes, unnecessary-level draining, and total down-compaction bytes.

`ComputeCompactionScore()` also triggers several sub-computations:
- `ComputeFilesMarkedForCompaction()` -- files with `marked_for_compaction` flag
- `ComputeFilesMarkedForPeriodicCompaction()` -- files older than `periodic_compaction_seconds`
- `ComputeExpiredTtlFiles()` -- files past TTL expiration
- `ComputeBottommostFilesMarkedForCompaction()` -- bottommost files with reclaimable sequence numbers
- `ComputeFilesMarkedForForcedBlobGC()` -- files needing blob garbage collection
- `EstimateCompactionBytesNeeded()` -- estimated bytes to reach target sizes

## L0 File Ordering

L0 files can have overlapping key ranges unlike L1+. They are sorted by `epoch_number` in decreasing order (newest first) for correct read ordering. The comparator (NewestFirstByEpochNumber) uses epoch_number descending, then fd.largest_seqno descending as a tiebreaker for files with the same epoch (e.g., after epoch number recovery).

For levels L1+, files are sorted by key range and are non-overlapping, enabling binary search via `FindFile()`.

## File Lookup

- **L1+ levels**: Binary search on `LevelFilesBrief.files[]` using `FindFile()`, which returns the smallest index where the file's largest key >= the search key.
- **L0 level**: Must check all files since key ranges overlap.
- **By file number**: `GetFileLocation()` and `GetFileMetaDataByNumber()` use the `file_locations_` hash map for O(1) lookup.
- **Overlap queries**: `GetOverlappingInputs()` returns files overlapping a key range, with optional expansion to include files that overlap each other.

## Dynamic Level Bytes

When `level_compaction_dynamic_level_bytes` is enabled, `CalculateBaseBytes()` computes `base_level_` -- the highest level that should contain data. Levels below `base_level_` should be empty. The `lowest_unnecessary_level_` tracks levels that are being drained.

The `level_multiplier_` field stores the actual multiplier between levels, which may differ from `max_bytes_for_level_multiplier` (see `ColumnFamilyOptions` in `include/rocksdb/advanced_options.h`) when dynamic level bytes adjusts it.

## Bottommost Files

`GenerateBottommostFiles()` identifies files whose key range does not overlap with any file at a lower level. `ComputeBottommostFilesMarkedForCompaction()` then marks those that contain duplicate key versions no longer protected by snapshots, making them eligible for space-reclaiming compaction.

For column families with user-defined timestamps, a file's largest timestamp must also be below `full_history_ts_low` before it can be marked.

## Consistency Checks

When `force_consistency_checks` is true (default: true; see `ColumnFamilyOptions` in `include/rocksdb/advanced_options.h`), `VersionBuilder::SaveTo()` runs `CheckConsistency()` to validate file ordering and overlap constraints. In debug builds these checks always run regardless of the option. See 05_version_builder.md for the split between Apply() and SaveTo() validation.
