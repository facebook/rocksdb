# Version Compatibility

**Files:** `include/rocksdb/version.h`, `db/version_edit.h`, `util/build_version.cc`

## Version Macros

RocksDB version is defined in `include/rocksdb/version.h`:

| Macro | Description |
|-------|-------------|
| `ROCKSDB_MAJOR` | Major version number |
| `ROCKSDB_MINOR` | Minor version number |
| `ROCKSDB_PATCH` | Patch version number |
| `ROCKSDB_VERSION_INT` | Combined integer: `major * 1000000 + minor * 1000 + patch` |
| `ROCKSDB_VERSION_GE(a, b, c)` | Compile-time check: current version >= a.b.c |

`GetRocksVersionAsString()` returns the version as a human-readable string (e.g., "11.1.0"). `GetRocksBuildProperties()` returns build metadata including git hash and build date.

In the main development branch, the version macros reflect the *next* planned release version, not the current released version.

## MANIFEST Forward Compatibility

RocksDB maintains forward compatibility of the MANIFEST format through two mechanisms:

### Top-Level Tag Masking

Tags in the `Tag` enum with `kTagSafeIgnoreMask` (bit 13) set are forward-compatible. When an older version encounters an unrecognized tag with this bit set, it skips the record safely. Tags without this bit cause a `Corruption` error if unrecognized.

Forward-compatible top-level tags include:
- `kDbId` -- database identifier
- `kWalAddition2`, `kWalDeletion2` -- WAL lifecycle tracking
- `kFullHistoryTsLow` -- UDT history watermark
- `kPersistUserDefinedTimestamps` -- UDT persistence flag
- `kSubcompactionProgress` -- resumable subcompaction state

### NewFile4 Custom Tag Masking

Per-file custom tags within `kNewFile4` records use a separate mask: `kCustomTagNonSafeIgnoreMask` (bit 6). Tags *without* this bit set can be safely ignored by older versions. Tags *with* this bit set (like `kPathId`) cause a failure if unrecognized.

Safely-ignorable custom tags include: `kNeedCompaction`, `kOldestBlobFileNumber`, `kTemperature`, `kUniqueId`, `kEpochNumber`, `kCompensatedRangeDeletionSize`, `kTailSize`, and `kUserDefinedTimestampsPersisted`.

## Downgrade Safety

When downgrading RocksDB to an older version, the MANIFEST can still be read as long as:
1. No non-ignorable tags are present that the older version doesn't understand
2. The older version supports the `kNewFile4` format (introduced as a divergence from open-source LevelDB)

Features that prevent downgrade:
- `kPathId` custom tag -- requires version awareness of multi-path storage
- Any future tag without the safe-ignore mask

## Compatibility Between Releases

RocksDB follows these compatibility principles:

- **Backward compatibility**: Newer versions can open DBs generated by all previous releases under normal configuration
- **Forward compatibility**: An older release can open DBs generated by releases several minor versions higher, as long as no new features are explicitly enabled that the older version doesn't support. This typically holds for a year or more of releases.
- **Patch releases** (e.g., 9.0.0 to 9.0.1): Fully compatible, no format changes
- **Minor releases** (e.g., 9.0 to 9.1): Forward-compatible MANIFEST format; new features use safe-ignore tags
- **Major releases**: May introduce breaking format changes, but typically with explicit migration support

The `kMinLogNumberToKeepHack` is an example of maintaining compatibility: rather than adding a new top-level tag (which would break old versions), `min_log_number_to_keep` was encoded inside `kNewFile4` custom tags where the safe-ignore mechanism applies.

Note: Changing the comparator or compaction style makes a DB unopenable even with the same code version. For compaction style changes, the migration utility at `include/rocksdb/utilities/option_change_migration.h` provides a helper.

## Options File

RocksDB materializes current DB options to an OPTIONS file in the DB directory. Users can read back options from this file to ensure compatible opening when upgrading or downgrading. However, the OPTIONS file parser does not share the same forward-compatibility guarantee as the MANIFEST format -- a new option added in a newer release (even with its default value) may produce a record that older versions cannot parse.
