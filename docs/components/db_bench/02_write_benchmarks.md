# Write Benchmarks

**Files:** `tools/db_bench_tool.cc`

## Write Benchmark Types

| Benchmark | WriteMode | fresh_db | Description |
|-----------|-----------|----------|-------------|
| `fillseq` | SEQUENTIAL | yes | Sequential key insertion. Best-case for LSM (no memtable sorting overhead) |
| `fillrandom` | RANDOM | yes | Random key insertion. Measures memtable sorting and compaction overhead |
| `filluniquerandom` | UNIQUE_RANDOM | yes | Shuffled unique keys. Single-threaded only. No duplicate keys |
| `fillbatch` | SEQUENTIAL | yes | Like `fillseq` with `entries_per_batch_` forced to 1000 |
| `overwrite` | RANDOM | no | Random writes on existing DB. Same as `fillrandom` but reuses current DB state. Users need a prior load phase or `--use_existing_db=1` to guarantee actual overwrites of preexisting keys |
| `fillsync` | RANDOM | yes | Durable writes with `sync=true`. Operates on `num/1000` keys |
| `fill100K` | RANDOM | yes | 100KB values. Operates on `num/1000` keys |
| `fillseqdeterministic` | SEQUENTIAL | yes | Fills levels with deterministic LSM shape. Requires `--disable_auto_compactions=1` |
| `filluniquerandomdeterministic` | UNIQUE_RANDOM | yes | Like `filluniquerandom` with deterministic LSM shape |
| `fillanddeleteuniquerandom` | UNIQUE_RANDOM | yes | Like `filluniquerandom` but supports interleaved deletions via `--disposable_entries_*` flags |

## WriteMode Enum

The `WriteMode` enum controls key generation in `DoWrite()`:

| Mode | Behavior |
|------|----------|
| `RANDOM` | Keys generated by `rand()` mod `FLAGS_num` |
| `SEQUENTIAL` | Keys generated sequentially: 0, 1, 2, ... |
| `UNIQUE_RANDOM` | Pre-shuffled permutation of `[0, num-1]`. Guarantees no duplicates. Single-threaded only |

## DoWrite Flow

The core write path `DoWrite()` works as follows:

Step 1: Create a `KeyGenerator` for the selected `WriteMode`. For `UNIQUE_RANDOM`, this pre-generates a shuffled permutation of keys.

Step 2: Create a `Duration` instance with `max_seconds` (from `--duration`, only for RANDOM mode) and `max_ops`.

Step 3: Main loop until `duration.Done()`:
- Generate next key via `KeyGenerator`
- Generate value via `RandomGenerator::Generate()` (controlled by `--compression_ratio`)
- Accumulate operations in a `WriteBatch` (even when `entries_per_batch_` is 1) and write via `DB::Write()`. The only exception is stacked BlobDB, which calls `blobdb->Put()` or `blobdb->PutWithTTL()` directly
- If `--user_timestamp_size > 0`, timestamps are injected into the batch via `WriteBatch::UpdateTimestamps()`
- Call `FinishedOps()` to record latency
- Apply write rate limiting via `SharedState::write_rate_limiter`

Step 4: If `--writes_per_range_tombstone > 0`, periodically insert `DeleteRange` tombstones during the write loop.

Step 5: Report found/written counts and byte throughput.

## Key Generation

Keys are generated by `GenerateKeyFromInt()` which formats an integer key into a fixed-size byte buffer:

- If `--keys_per_prefix > 0`: key format is `[prefix][key_body + padding]`. The prefix is computed as `v % (num_keys / keys_per_prefix)`, so prefixes rotate across the key space rather than being assigned by contiguous groups. The prefix is written as big-endian bytes (up to 8 bytes), with any remaining prefix bytes padded with `'0'`. The key body is also binary bytes, not decimal text
- If `--keys_per_prefix == 0` (default): key format is `[key_number + zero_padding]` padded to `--key_size` bytes

Note: User-defined timestamps are injected into the `WriteBatch` via `WriteBatch::UpdateTimestamps()`. Only 8-byte timestamps are supported.

## Value Generation

`RandomGenerator` pre-generates a buffer (at least 1MB) of compressible data using `test::CompressibleString()`. Each call to `Generate()`:

1. Selects the value size from the configured distribution (fixed, uniform, or normal)
2. Returns a `Slice` pointing into the pre-generated buffer at the current offset
3. If `--same_value_percentage > 0`, returns the same fixed prefix with configured probability

The `--compression_ratio` flag controls compressibility: 0.5 means data compresses to approximately 50% of its original size. A ratio of 1.0 produces incompressible data.

## Overwrite and Disposable Entry Modes

The `filluniquerandom` benchmark supports two advanced modes:

**Overwrite mode** (`--overwrite_probability`): For each write, probabilistically overwrites a key from a sliding window of recently-inserted keys (`--overwrite_window_size`).

**Disposable/persistent entries** (`--disposable_entries_batch_size`): Simulates a workflow where batches of temporary keys are inserted, followed by persistent result keys, then the temporary keys are deleted. Controlled by:
- `--disposable_entries_batch_size`: number of temporary keys per batch
- `--disposable_entries_delete_delay`: delay before issuing deletes
- `--persistent_entries_batch_size`: number of persistent keys inserted before deletes
- Separate value sizes for disposable vs persistent entries

Note: Disposable entries mode is incompatible with `DeleteRange` and overwrite mode.

## Column Family Rotation

When `--num_hot_column_families > 0` and `--num_column_families > 1`:
- Only `num_hot` CFs are active at any time
- After `ops_per_stage` operations, new CFs are created via `DBWithColumnFamilies::CreateNewCf()`
- CF selection uses either uniform random or weighted distribution (`--column_family_distribution`)
