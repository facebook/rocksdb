# Core-Local Implementation

**Files:** `monitoring/statistics_impl.h`, `monitoring/statistics.cc`, `util/core_local.h`, `monitoring/histogram.h`, `monitoring/histogram.cc`

## Core-Local Architecture

`StatisticsImpl` uses core-local storage to avoid cache line contention on the hot path. Each CPU core has its own `StatisticsData` struct containing all tickers and histograms. This design scales to high core counts without cross-core synchronization during recording.

The `CoreLocalArray<StatisticsData>` template (see `util/core_local.h`) allocates an array of `StatisticsData` entries, one per core. The array size is the smallest power of two greater than or equal to `std::thread::hardware_concurrency()` (minimum 8). Each access maps the current physical core ID to an array index using `port::PhysicalCoreID()`.

## StatisticsData Layout

The `StatisticsData` struct (see `monitoring/statistics_impl.h`) is cache-line aligned to prevent false sharing:

- `tickers_[INTERNAL_TICKER_ENUM_MAX]` -- array of `std::atomic_uint_fast64_t` for all tickers
- `histograms_[INTERNAL_HISTOGRAM_ENUM_MAX]` -- array of `HistogramImpl` for all histograms
- Padding to ensure each `StatisticsData` is a multiple of `CACHE_LINE_SIZE` bytes

The struct uses `ALIGN_AS(CACHE_LINE_SIZE)` and custom `operator new` with `port::cacheline_aligned_alloc()` to guarantee alignment.

## Recording Path (Hot Path)

Step 1: `recordTick()` checks `StatsLevel` -- if stats are disabled (`<= kExceptTickers`), returns immediately.

Step 2: `per_core_stats_.Access()` retrieves the `StatisticsData` for the current core using `port::PhysicalCoreID()`. If the core ID is unavailable (returns negative), a random index is chosen.

Step 3: The ticker is incremented with `fetch_add(count, std::memory_order_relaxed)`. No lock is held.

Step 4: If an inner `Statistics` object exists (`stats_`), the recording is forwarded to it.

For histograms, `recordInHistogram()` follows the same pattern: check `StatsLevel`, access core-local data, call `HistogramImpl::Add()` on the core-local histogram.

**Note:** The recording path is lock-free. The only synchronization is the atomic increment with relaxed memory ordering.

## Reading Path (Aggregation)

Reading a ticker value requires aggregating across all cores:

Step 1: Acquire `aggregate_lock_` (a `port::Mutex`).

Step 2: Iterate over all core slots via `per_core_stats_.AccessAtCore(core_idx)`.

Step 3: Sum the per-core ticker values.

The same pattern applies to histograms: `getHistogramImplLocked()` creates a new `HistogramImpl`, then merges each core's histogram into it.

**Note:** Reading is O(num_cores) and requires a lock. This is acceptable because reads are infrequent relative to recordings.

## Histogram Internals

`HistogramStat` (see `monitoring/histogram.h`) uses a fixed set of 109 buckets managed by `HistogramBucketMapper`. Bucket boundaries are generated by repeatedly multiplying by 1.5 and rounding to two significant digits, providing finer granularity for small values and coarser granularity for large values.

Each `HistogramStat` maintains:
- `min_`, `max_` -- range tracking (atomic)
- `num_` -- sample count (atomic)
- `sum_` -- sum of all values for average calculation (atomic)
- `sum_squares_` -- sum of squares for standard deviation calculation (atomic)
- `buckets_[109]` -- value distribution across 109 buckets (each atomic)

Percentile computation uses linear interpolation within the bucket containing the target percentile rank. Percentiles are approximate -- exact accuracy depends on bucket granularity.

## Reset Semantics

`Reset()` acquires `aggregate_lock_` and:
1. Sets all tickers to zero across all cores
2. Clears all histograms on all cores

`getAndResetTickerCount()` atomically exchanges each core's ticker value with zero (using `std::atomic::exchange`) and returns the sum.

## Internal Tickers and Histograms

`TickersInternal` and `HistogramsInternal` enums (see `monitoring/statistics_impl.h`) extend the public enums with internal-only metrics. Currently both ranges are empty (no internal-only tickers or histograms defined between `INTERNAL_*_START` and `INTERNAL_*_MAX`), but the mechanism exists for adding internal metrics without exposing them in the public API.
