# Compressed Secondary Cache

**Files:** `cache/compressed_secondary_cache.h`, `cache/compressed_secondary_cache.cc`, `include/rocksdb/cache.h`

## Overview

`CompressedSecondaryCache` is the built-in `SecondaryCache` implementation that stores blocks in memory using an internal LRU cache, compressing most entries with LZ4 by default. It provides a memory-efficient second tier that trades CPU cycles for reduced memory footprint. The internal backing store is a standard LRU cache, making it effectively a "cache within a cache."

## Compression Behavior

- Default compression: LZ4 (configurable via `compression_type` in `CompressedSecondaryCacheOptions`, see `include/rocksdb/cache.h`)
- Entries with roles in `do_not_compress_roles` (default: `{kFilterBlock}`) are stored uncompressed, since filter blocks are essentially non-compressible
- If the compressed output is not smaller than the original, the entry is stored uncompressed
- Compression and decompression use the V2 compression framework. The constructor obtains `Compressor` and `Decompressor` objects from `GetBuiltinV2CompressionManager()` (see `include/rocksdb/advanced_compression.h`)

## Tagged Value Format

Values stored in the internal LRU cache have a 2-byte tag header before the actual data:

| Byte | Field | Description |
|------|-------|-------------|
| 0 | Source `CacheTier` | Which tier is responsible for compression/decompression |
| 1 | `CompressionType` | Compression algorithm used (generated by this cache if source is `kVolatileCompressedTier`, preserved from input otherwise) |

When `enable_custom_split_merge` is false, the tagged value is prefixed with a varint64 length (for `GetLengthPrefixedSlice`). When true, the tagged value is split into `CacheValueChunk` linked-list nodes without a length prefix.

The source tier field distinguishes between:
- `kVolatileCompressedTier`: entry was compressed by this secondary cache (from primary cache eviction)
- `kVolatileTier`: entry was already compressed externally (from SST block via `InsertSaved`)

## Insertion Flow

### From Primary Cache Eviction (`Insert`)

Step 1: Check `force_insert`. If false, apply the two-hit admission protocol via `MaybeInsertDummy()` (see chapter 3).

Step 2: Call `InsertInternal()` with `source = kVolatileCompressedTier` and `from_type = kNoCompression` (the primary cache holds uncompressed blocks).

Step 3: Serialize the object using `helper->size_cb` and `helper->saveto_cb` into a buffer.

Step 4: If compression is enabled and the entry's role is not in `do_not_compress_roles`, compress the data. If the compressed output is not smaller, store uncompressed.

Step 5: Write the 2-byte tag (source tier + compression type) into the header.

Step 6: Insert into the internal LRU cache, either as a `CacheValueChunk` chain (split/merge mode) or as a length-prefixed allocation.

### From SST Block (`InsertSaved`)

Step 1: Validate inputs: `InsertSaved` checks three conditions sequentially with early returns: (a) `source != kVolatileCompressedTier` -- this check includes a debug assertion, so in debug builds it will abort rather than silently returning; (b) `type != kNoCompression`; (c) `!enable_custom_split_merge`. If any condition fails, the call returns without inserting.

Step 2: Apply two-hit admission via `MaybeInsertDummy()`.

Step 3: Call `InsertInternal()` with the provided compression type and source tier, preserving the existing compression.

## Lookup Flow

Step 1: Look up the key in the internal LRU cache.

Step 2: If not found, return `nullptr`.

Step 3: If found but value is `nullptr` (dummy entry), release the handle and record `COMPRESSED_SECONDARY_CACHE_DUMMY_HITS`. Return `nullptr`.

Step 4: Extract the tagged data. If `enable_custom_split_merge`, merge `CacheValueChunk` chain into a contiguous string. Otherwise, read via `GetLengthPrefixedSlice`.

Step 5: Parse the 2-byte tag to get source tier and compression type.

Step 6: If `source == kVolatileCompressedTier` and compression type is not `kNoCompression`, decompress the data using the `Decompressor`.

Step 7: Call `helper->create_cb` to reconstruct the cache object from the (decompressed) data.

Step 8: Handle `advise_erase`: if true, erase the real entry and insert a dummy. If false, keep the entry and set `kept_in_sec_cache = true`.

Step 9: Return a `CompressedSecondaryCacheResultHandle` wrapping the result. This handle is always immediately ready (`IsReady()` returns `true`; `Wait()` is a no-op).

## Jemalloc-Friendly Chunking

When `enable_custom_split_merge = true`, compressed values are split into chunks sized to fit jemalloc allocation bins. The target bin sizes are: 128, 256, 512, 1024, 2048, 4096, 8192, and 16384 bytes (see `malloc_bin_sizes_` in `compressed_secondary_cache.h`).

`SplitValueIntoChunks()` allocates each chunk to fit the nearest jemalloc bin size, reducing internal fragmentation. Each chunk is a `CacheValueChunk` struct containing a `next` pointer, a `size` field, and the data payload. The charge to the internal LRU is the sum of actual allocation sizes (not the logical data sizes).

`MergeChunksIntoValue()` walks the linked list and concatenates all chunk data into a single contiguous string on lookup.

Note: the split/merge mode is not compatible with `InsertSaved()` (tiered cache warming). When `enable_custom_split_merge` is true, `InsertSaved` is a no-op.

## Capacity Management

- `SetCapacity(capacity)`: adjusts the internal LRU cache capacity. Setting to 0 effectively disables the cache (all lookups return `nullptr` via the `disable_cache_` flag).
- `Deflate(decrease)`: temporarily reduces effective capacity by reserving space via `ConcurrentCacheReservationManager`. Used by `CacheWithSecondaryAdapter` for proportional reservation distribution.
- `Inflate(increase)`: restores capacity reduced by a prior `Deflate`.

Both `Deflate` and `Inflate` are lighter-weight than `SetCapacity` because they operate via cache reservation rather than redistributing capacity across all shards.

Note: when `ROCKSDB_MALLOC_USABLE_SIZE` is defined and `enable_custom_split_merge` is false, the charge for entries in the internal LRU is based on `malloc_usable_size()` rather than the logical data size. This can cause the cache to use slightly different effective capacity than expected, depending on the allocator's per-allocation overhead.

When decompression fails during `Lookup`, the entry is erased from the internal LRU cache and `nullptr` is returned. The caller (adapter) treats this as a cache miss and falls back to reading from storage. The original compressed entry is lost. In debug builds, assertions will fire on decompression failure.

## Perf Context Counters

| Counter | Description |
|---------|-------------|
| `compressed_sec_cache_insert_dummy_count` | Dummy entries inserted (first touch) |
| `compressed_sec_cache_insert_real_count` | Real entries inserted (second touch) |
| `compressed_sec_cache_uncompressed_bytes` | Uncompressed bytes before compression |
| `compressed_sec_cache_compressed_bytes` | Bytes after compression |

Ticker: `COMPRESSED_SECONDARY_CACHE_HITS` and `COMPRESSED_SECONDARY_CACHE_DUMMY_HITS` in `Statistics`.
